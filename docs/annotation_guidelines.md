# 简体中文 PII 标注指南 v1.0

状态：**待业务、安全与数据合规评审**。本文落实阶段 A 的静态标注规范，不表示现网
ERNIE、内部金标或生产数据已经取得或评测。

机器可读标签定义以 `src/pii_zh/taxonomy/taxonomy.yaml` 为准；本文解释如何画边界和
处理歧义。任何标签变更必须同时更新 taxonomy 版本、Presidio 映射、数据合同、转换器
和回归测试。

## 1. 基本单位与边界

1. 真值是原始字符串上的字符 span，使用 Python 字符串索引语义和左闭右开区间
   `[start, end)`；必须满足 `text[start:end] == entity.text`。
2. 索引基于解码后的 Unicode 字符，不是 UTF-8 字节、UTF-16 code unit、可视字形或
   tokenizer token。换行、emoji、全角字符、组合字符和零宽字符都占据其实际 Python
   索引位置。
3. 标注最小且完整的实体值。字段名、提示词、句末标点、引号和外围空白通常不进入
   span；实体内部有意义的空格、短横线、括号或换行可以进入 span。
4. 不修正文案。OCR/ASR 错字、全半角、大小写和分隔符按原文精确标注，并用元数据
   记录噪声类型。
5. 原始标注可以保存嵌套或重叠 span。训练 BIO 视图是后续确定性压平的派生产物，
   不能覆盖原始标注。

## 2. 三层职责

- **检测层**回答文本片段是什么，使用本 taxonomy 的原子标签。
- **风险层**记录 `T0`、`T1`、`T2`，用于评测和策略输入，不等同于法律结论。
- **策略层**结合租户、场景、允许列表、校验器和用途决定是否及如何遮盖。标注员不因
  某客户当前“不遮盖”而漏标真实实体，也不因看到普通日期/组织名就扩大 PII 范围。

## 3. 核心标签边界

### 3.1 姓名、地址、日期和联系方式

- `PERSON_NAME`：标注自然人姓名、别名或明确指向自然人的称呼值；排除先生/女士等
  纯头衔，除非它是固定别名的一部分。公司、产品、游戏角色和无法指向自然人的普通词
  作为困难负样本。中英文混合姓名保留内部空格。
- `ADDRESS`：标注足以定位个人的连续详细地址短语，通常包含区县以下细节；只出现
  国家、省、市、景点或组织办公地址时默认不标。字段中省市区、街道、楼栋、单元、房号
  连续出现时合为一个 span；跨字段离散出现时分别标注并保留同一关系组元数据。
- `DATE_OF_BIRTH`：只有“出生、生日、生于”等上下文明确时标注日期值。合同日期、
  订单日期和日志时间标为辅助 `GENERIC_DATE`（若实验开启）或保持非 PII。
- `PHONE_NUMBER`：标注手机、座机和分机值，包括值内部的 `+86`、括号、空格、短横线
  和分机分隔；排除字段名及末尾标点。公共客服电话标为辅助 `PUBLIC_HOTLINE` 或负样本。
- `EMAIL_ADDRESS`：标注完整邮箱值；外围尖括号/引号仅在确属地址语法的一部分时纳入。

### 3.2 高风险结构化标识

- `CN_RESIDENT_ID`、`BANK_CARD_NUMBER`：标注原文完整候选，包括内部合法分隔。校验位
  失败的近似值仍可作为困难负样本，但不得因为“长得像”就标为真值；记录校验结果。
- `PASSPORT_NUMBER`、`DRIVER_LICENSE_NUMBER`、`SOCIAL_SECURITY_NUMBER`：需要格式或
  明确字段上下文。通用订单号、合同号、随机长数字不得强行映射。
- `BANK_ACCOUNT_NUMBER`：仅在银行账户/开户账号语义成立时使用；卡号优先使用更具体的
  `BANK_CARD_NUMBER`。
- `MEDICAL_RECORD_NUMBER`：病历号、就诊号、住院号等个人医疗记录标识。普通科室、
  诊断文字不属于本标签。
- `GEO_COORDINATE`：标注明确经纬度值或坐标对；普通门牌地址使用 `ADDRESS`。
- `SECRET`：标注密码、API key、访问 token 或私钥片段等安全敏感值。标注环境不得导入
  真实可用 secret；真实泄漏应走安全事件流程而不是进入语料。

### 3.3 个人账号与设备标识

- `WECHAT_ID`、`QQ_NUMBER`、`ALIPAY_ACCOUNT`、`USERNAME`：必须有账号类型或登录语义；
  昵称与自然人姓名同时成立时保留原始重叠候选，训练压平时按优先级处理。
- `EMPLOYEE_ID`、`STUDENT_ID`：必须有员工/学生上下文或经批准的租户格式。它们可以
  按场景落在 `T1/T2`，但标签类型不随策略改变。
- `DEVICE_ID`：IMEI、硬件序列号等稳定设备标识。型号、软件版本和 trace ID 是负样本。
- `IP_ADDRESS`、`MAC_ADDRESS`：标注完整地址值；端口不属于 IP span。格式校验失败的
  相似字符串作为困难负样本。
- `VEHICLE_LICENSE_PLATE`：标注完整车牌值；仅有车型、车系或车辆品牌时不标。

## 4. 扩展和辅助标签

`HEALTH_INFORMATION`、`FINANCIAL_INFORMATION`、`BIOMETRIC_INFORMATION`、
`RELIGIOUS_BELIEF`、`PRECISE_MOVEMENT`、`MINOR_INFORMATION`、`AGE`、`GENDER`、
`OCCUPATION` 属于 v1.1 扩展，默认关闭。启用前必须补充正反例、最小边界、业务需求、
标注一致性和独立评测，不得把法规名词直接当成无边界的全量遮盖指令。

`ORDER_ID`、`TRANSACTION_ID`、`INVOICE_CODE`、`PRODUCT_CODE`、`ORGANIZATION`、
`JOB_TITLE`、`GENERIC_DATE`、`PUBLIC_HOTLINE`、`ORG_ADDRESS` 是可选的训练混淆标签，
默认不作为推理输出。它们的作用是显式监督高相似负样本。

## 5. 重叠、嵌套和冲突

原始层保留全部有依据的 span，并记录冲突。生成平面 BIO 训练视图时依次优先：

1. 高风险、格式明确的具体标识；
2. 具体账号类型；
3. 姓名、地址和联系方式；
4. 扩展敏感属性；
5. 辅助混淆标签。

相同边界保留更具体类型；包含关系保留高优先级类型。转换器必须输出每类冲突数、丢弃
的派生标签及原因，禁止静默删除原始标注。无法确定类型或边界时标为待裁决，不猜测。

## 6. 困难负样本

无 PII 文档和高相似负样本必须被显式收集：订单/合同/快递/发票号、商品/版本/端口号、
普通日期/金额、企业统一社会信用代码、公司地址、公共客服电话、无效身份证/银行卡、
已脱敏值、hash/UUID/trace ID，以及只有“身份证号”等字段名而没有值的文本。

负样本同样需要来源、许可、分组和复核。不得把“校验失败”当作放弃人工复核的通用理由，
因为真实文本可能包含 OCR/ASR 噪声。

## 7. 标注流程与质量控制

1. 只在获批的隔离环境中处理真实业务文本；公开开发和单元测试只用合成数据。
2. 预标注可以来自现有 Presidio、ERNIE 和规则，但至少保留一部分无预标注样本以测量
   锚定偏差。本仓库尚无现网 ERNIE 基线，不能声称已有预标注质量。
3. 至少 15%–20% 样本由两人独立标注；高级标注员裁决冲突。
4. 记录边界一致率、类型一致率、标注员间 span-F1、每类数量和裁决原因。
5. 每次指南变更提升版本并重新评审受影响样本；测试集冻结后不得用于提示、规则、阈值
   或标签边界调试。
6. 导出前运行 span 等式、范围、重复/重叠、标签、风险层、来源、许可和分组校验。
7. 报告只记录计数、比例、哈希和脱敏示例，不记录原始 PII。
